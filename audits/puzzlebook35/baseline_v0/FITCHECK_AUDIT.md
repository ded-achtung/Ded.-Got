## Findings

### `who` (name) — informationally useless on this corpus

Matches in **30 / 30** chunks (100%). 381 total matches, 96 unique.

Top hits are not names: `Выход`, `Вход`, `Python`, `Если`, `Пример`, `Функция`, `Имя`, `Тип`, `Заготовка`, `Здесь`. These come from table column headers, example labels (`Вход:`/`Выход:`), Python identifiers, sentence-initial words after `.`, and reserved words.

Implication: **`fit_status=match` is the foregone outcome for any intent=who question on this corpus.** `mismatch` cannot be reached. Q7's `match → hit` was not a coincidence — it is structurally guaranteed by the corpus + this regex.

Synthetic probes confirm: **5 / 7 negative-control inputs over-match** (sentence-initial common words, language/product names, mid-sentence capitalized tokens).

### `how_many` (number) — too broad to discriminate

Matches in **22 / 30** chunks (73%). 428 total matches.

Top hits are not counts: `1`, `2`, `3`, … — these are mostly list markers (`1. Первый…`), version strings (`3.10`), example data literals (`[1, 2, 3]`), and ASCII codes inside the Morse-code task. Almost any prose chunk and every code-bearing chunk match.

Implication: **`fit_status=match` happens for ~73% of chunks regardless of question semantics.** When Q5 or Q14 returns `match`, this is not evidence that the chunk contains a count answer; it is evidence that the chunk contains *any* digit. Distinguishing 'count answer' from 'incidental digit' is not done.

Synthetic probes: **2 negative-control over-matches** (list markers, version numbers).

### `when` (year) — tight on this corpus, but coincidentally

Matches in **0 / 30** chunks. Q22's `fit_refuse` works because the corpus simply does not contain any `19xx`/`20xx` token in pp.12–38.

Synthetic probes: **2 over-matches** on year-shaped non-years (counts, ranges, decades), and **1 false-negative** on year-with-suffix.

Implication: this regex is **the only one of the three that could plausibly refuse** — but only when no year-shaped number appears for any reason. On a corpus with code containing magic numbers (e.g., `port=1999`, `BUFFER_SIZE=2048`), it would lose this property immediately. Q22 audit row's machine-verifiable check (NOTES §3) is more robust than the bare regex because it operates on downstream-effective-k content with explicit hit listing — but the regex itself is brittle.

## Recommendations (do not implement here — record only)

1. **`who` regex is unfit for purpose** on this corpus. Before extending fit_check or relying on `match → hit` for any intent=who question, the rule needs a fundamentally different design: stop-word filter for sentence-initial common words, NER, or context-aware patterns (e.g., 'имя автора' / 'написал' / 'by' as triggers).
2. **`how_many` regex needs context.** Distinguishing count answers from list markers / version strings / data literals requires either question-conditioned matching (does any number co-occur with the question's noun?) or a stop-list of known non-counts.
3. **`when` regex is acceptable** as-is for refuse-on-no-year, but the match→hit branch is brittle (year-shaped count → false hit). Could be tightened with adjacency context (`год`/`году`/`year` within ±N tokens).
4. **Naively cloning these regexes for `what`/`why`/`how`** would inherit the worst properties of #1 and #2: regexes that match almost everything in any prose corpus. The fit_check extension needs a different model class, not more of the same.
5. **Q7 audit-row prediction reframed:** the `lexical_pattern_overmatch` failure mode (NOTES §5) is not just a `who` problem — it is a general property of bare regex fit_check. Two of three v0 rules exhibit it. Only one (`when`) avoids it, by accident of corpus content.
